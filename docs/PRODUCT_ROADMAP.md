## 一、项目选题报告

**业务场景**  
互联网热搜榜单是反映社会注意力的重要窗口，但各平台热搜分散、历史数据难以追溯、缺乏面向个人和小团队的轻量分析工具。现有产品要么只做聚合展示，要么是面向大企业的高价营销/舆情系统，中间存在明显空白。

**潜在需求**  
C端用户想快速了解"今天大家都在关注什么"、回溯历史热搜、获取选题灵感；B端用户（自媒体、小团队、中小企业）需要低成本的热搜分析工具，了解品牌曝光、行业趋势、平台特征。

**主要功能**  
多平台热搜聚合展示、历史热搜搜索查询、话题跨平台传播分析、品牌热搜追踪、AI驱动的热点问答与洞察生成。

**设计目标**  
- 基本：稳定的数据采集与存储，提供热搜聚合页和历史搜索功能  
- 进阶：跨平台话题聚合与传播分析，品牌/行业追踪面板  
- 挑战：基于历史数据的知识库构建，AI实现智能洞察与报告生成

## 二、数据本质再认识

### 2.1 热搜词条的本质特征

热搜词条不是传统意义上的"内容"，而是**社会注意力的聚合点**。理解这一本质是产品设计的基础。

| 特征维度 | 具体描述 | 对产品设计的影响 |
|----------|----------|------------------|
| **极简性** | 5-20个字的短语，如"某某事件"、"某某回应" | 信息密度极低，需要补充背景才能理解完整含义 |
| **涌现性** | 不是被"发布"的，而是由用户行为"涌现"的 | 没有传统意义的"发布时间"，只有"进入榜单时间" |
| **聚合性** | 代表大量用户在同一时间的关注焦点 | 反映的是群体注意力，而非个体内容 |
| **周期性** | 生命周期通常在几小时到几天 | 历史数据价值大于实时数据 |
| **跨平台性** | 同一事件会在多个平台发酵 | 跨平台分析是核心差异化能力 |

### 2.2 热搜与内容的本质区别

| 维度 | 传统内容 | 热搜词条 |
|------|----------|----------|
| 来源 | 作者主动发布 | 用户行为涌现 |
| 时间属性 | 有明确的发布时间 | 只有"进入榜单"的时间点 |
| 内容形态 | 完整的文章/视频/图片 | 极简的短语 |
| 价值维度 | 内容本身的价值 | 注意力聚合的价值 |
| 分析方法 | 内容分析 | 趋势分析、传播分析、历史对比 |

### 2.3 数据采集的真实约束

**硬性约束**：

| 约束 | 具体限制 | 影响 |
|------|----------|------|
| 采集频率 | GitHub Actions 最小间隔约5分钟，实际运行约每小时1-2次 | 无法获取分钟级数据 |
| API限制 | 部分平台有访问频率限制 | 无法提高采集频率 |
| 平台差异 | 9个平台的数据结构、更新频率不同 | 跨平台对比需要归一化处理 |
| 字段缺失 | 大部分平台不提供发布时间、作者信息 | 时间序列构建困难 |

**数据边界**：

```
我们能够获取的数据：
├─ crawled_at（爬取时间）- 精确到小时级
├─ rank（排名）- 每次快照的排名位置
├─ hot_value（热度值）- 每次快照的热度
├─ title（标题）- 热搜词条文本
├─ description（描述）- 部分平台提供
└─ platform_fields（平台特有字段）

我们无法获取的数据：
├─ 事件"开始"时间 - 热搜是涌现的，没有明确起点
├─ 内容"发布"时间 - 大部分平台不提供
├─ 精确的事件时间线 - 需要通过其他方式推断
└─ 用户行为数据 - 谁在搜索、为什么搜索
```

## 三、时间序列的真实约束与应对

### 3.1 时间序列数据的根本困境

**核心问题**：我们无法准确知道事件"真正"开始的时间。

```
时间线困境示意：

真实事件发展（未知）：
事件发生 → 开始传播 → 进入热搜 → 热度峰值 → 退出热搜
   ?          ?          ?           ?          ?
   │          │          │           │          │
   └──────────┴──────────┴───────────┴──────────┘
              我们完全不知道这段发生了什么

我们能观测到的：
爬取点1    爬取点2    爬取点3    爬取点4    爬取点5
   ●          ●          ●          ●          ●
   │          │          │          │          │
T+0小时    T+1小时    T+2小时    T+3小时    T+4小时
排名#15     排名#8     排名#2     排名#5     排名#20

→ 我们只能看到"离散的点"，无法还原"连续的线"
→ 我们不知道事件在T+0之前已经存在多久
```

### 3.2 时间序列洞察的准确性评估

**诚实的评估**：基于小时级快照的时间序列分析存在以下局限：

| 分析类型 | 准确性 | 局限性 |
|----------|--------|--------|
| 排名趋势 | 中等 | 只能看到离散点，无法捕捉快速变化 |
| 热度趋势 | 中等 | 同上，且不同平台热度不可直接比较 |
| 生命周期估算 | 较低 | 只能估算"在榜时间"，无法估算"真实生命周期" |
| 传播路径 | 较低 | 只能看到"哪个平台先出现"，无法确定真实传播顺序 |
| 峰值时间 | 较低 | 可能错过真正的峰值 |

### 3.3 可行的应对策略

**策略一：承认局限，明确边界**

```
正确的表述：
✓ "该话题在X点到Y点期间保持在榜单中"
✓ "该话题在观测期间排名上升了N位"
✓ "该话题首次出现在我们的数据中是X点"

错误的表述：
✗ "该话题的生命周期是X小时"（我们不知道真实生命周期）
✗ "该话题在X点达到峰值"（可能错过真实峰值）
```

**策略二：聚焦"相对时间"而非"绝对时间"**

```
我们可以准确分析的：
├─ 话题在榜单中的"相对位置变化"（上升/下降）
├─ 话题在不同平台的"出现顺序"（谁先谁后）
├─ 话题的"在榜时长"（从首次出现到最后出现）
├─ 话题的"排名区间"（最高排名、最低排名）
└─ 话题的"热度相对变化"（上升百分比、下降百分比）
```

**策略三：利用跨平台数据弥补单平台不足**

虽然单平台的时间分辨率有限，但跨平台数据可以提供更完整的视角：

```
跨平台时间线重建：

平台A（百度）:  ●─────●─────●─────○
               T+0   T+1   T+2   T+3

平台B（微博）:  ○─────●─────●─────●
               T+0   T+1   T+2   T+3

平台C（知乎）:  ○─────○─────●─────●
               T+0   T+1   T+2   T+3

综合判断：
- 话题可能在T+0之前就已经存在（平台A已进入榜单）
- 话题在T+1开始跨平台传播
- 话题在T+2达到最广泛的覆盖
```

**策略四：积累历史数据提升分析能力**

| 数据积累时长 | 可进行的分析 |
|--------------|--------------|
| 1天 | 单日热搜列表、基础排名 |
| 1周 | 周度趋势、话题持续时长分布 |
| 1月 | 月度热点回顾、平台特征对比 |
| 3月 | 季度趋势、周期性热点识别 |
| 1年 | 年度热点回顾、长期趋势分析 |

### 4.6 我们的差异化定位

**核心问题**：即时热榜已有历史查询，我们还有什么差异化？

**深度分析**：

| 维度 | 即时热榜 | 营销数据平台 | 舆情系统 | 我们 |
|------|----------|--------------|----------|------|
| 数据范围 | 热搜聚合 | 单平台深度 | 全网舆情 | **热搜+历史** |
| 历史深度 | 浅（近期） | 中等 | 深 | **深（长期积累）** |
| 跨平台 | 有 | 无/弱 | 有 | **有（9平台统一）** |
| AI能力 | 无 | 部分 | 部分 | **核心能力** |
| 目标客户 | C端 | 品牌方/MCN | 大企业 | **C端+小B端** |
| 价格 | 免费/低价 | 高 | 很高 | **低价** |

**真正的差异化**：

1. **历史数据深度**：即时热榜的历史查询深度有限，我们做长期积累
2. **AI洞察生成**：不只是查询，而是"理解+分析+洞察"
3. **跨平台统一视角**：9大平台统一格式，便于对比分析
4. **轻量低门槛**：面向个人和小团队，价格亲民

## 五、产品核心定位重新思考

### 5.1 定位的迭代

**初步定位**：热搜聚合 + 历史查询
- 问题：即时热榜已有此功能，差异化不足

**二次定位**：RAG+LLM智能分析
- 问题：这是技术手段，不是产品价值

**最终定位**：**热搜历史数据洞察平台**

核心价值主张：**"让历史热搜数据产生洞察"**

不是"聚合"（今日热榜已做），不是"监控"（舆情系统已做），而是**"洞察"**。

### 5.2 "洞察"的具体含义

```
查询 vs 洞察：

查询（即时热榜）：
用户："上周有哪些热搜？"
系统：返回上周热搜列表
→ 用户自己分析

洞察（我们）：
用户："最近科技领域有什么热点趋势？"
系统：
1. 检索近30天科技相关热搜
2. 分析热度变化、平台分布
3. 识别上升/下降趋势
4. 生成结构化洞察报告
→ 用户直接获得结论
```

### 5.3 AI功能设计

**功能一：热点问答（基础）**

```
用户提问示例：
├─ "最近一周科技领域有哪些热点？"
├─ "某某品牌最近上过热搜吗？表现如何？"
├─ "这个话题之前出现过吗？历史情况怎样？"
├─ "最近娱乐八卦有什么热点？"
└─ "帮我分析一下某某事件的热度趋势"

AI回答逻辑：
1. 理解用户意图（领域/品牌/话题/趋势）
2. 检索历史热搜数据
3. LLM生成结构化回答
4. 附上数据来源
```

**功能二：洞察报告生成（进阶）**

```
报告类型：
├─ 品牌热搜报告：品牌近期热搜表现分析
├─ 行业热点报告：某领域近期热点汇总
├─ 话题追踪报告：某话题的跨平台传播分析
└─ 时间段报告：某时间段的热搜回顾

报告结构：
├─ 概述：AI生成的摘要
├─ 数据：关键指标和图表
├─ 分析：AI生成的洞察
└─ 建议：AI给出的建议
```

**功能三：智能推荐（挑战）**

```
推荐场景：
├─ 相关话题推荐：基于语义相似度
├─ 趋势预测推荐：基于历史规律
├─ 行业热点推荐：基于用户关注领域
└─ 品牌关联推荐：基于实体识别
```

### 5.4 RAG架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                    RAG架构                                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  知识库构建                                                  │
│  ├─ 数据源：一道数据（热搜快照）+ 二道数据（AI增强）          │
│  ├─ 文档切分：按话题/时间/平台切分                           │
│  ├─ 向量化：使用Embedding模型生成向量                        │
│  └─ 索引：存储到向量数据库                                   │
│                                                             │
│  检索流程                                                    │
│  ├─ 用户提问 → 问题理解 → 向量化                            │
│  ├─ 向量检索 → 召回相关文档                                 │
│  ├─ 重排序 → 筛选最相关内容                                 │
│  └─ 上下文构建 → 组装Prompt                                 │
│                                                             │
│  生成流程                                                    │
│  ├─ Prompt + 上下文 → LLM                                   │
```

## 九、二道数据处理展望

### 9.1 二道数据的定位

二道数据的核心目标：**将"极简的热搜词条"转化为"可理解、可分析、可检索的信息单元"**。

### 9.2 处理流程

```
一道数据 → 二道数据：

输入（一道数据）：
{
  "title": "某某事件",
  "hot_value": 5000000,
  "platform": "weibo",
  "crawled_at": "2026-02-12 14:00:00"
}

输出（二道数据）：
{
  "title": "某某事件",
  "title_normalized": "某某事件",
  "category": "社会",
  "sub_category": "社会热点",
  "sentiment": "neutral",
  "sentiment_score": 0.6,
  "entities": {
    "persons": ["张三"],
    "organizations": ["某某公司"],
    "locations": ["北京"]
  },
  "keywords": ["事件", "某某", "回应"],
  "summary": "该事件涉及...",
  "embedding": [0.1, 0.2, ...],
  "cross_platform": {
    "platforms": ["weibo", "baidu", "zhihu"],
    "first_seen_platform": "weibo",
    "spread_pattern": "weibo → baidu → zhihu"
  }
}
```

### 9.4 LLM处理策略

| 处理类型 | 用途 | 模型选择 | 成本估算 |
|----------|------|----------|----------|
| 分类 | 话题类别 | GPT-3.5/通义 | 低 |
| 情感分析 | 情感倾向 | GPT-3.5/通义 | 低 |
| 实体识别 | 人名/机构 | GPT-3.5/通义 | 低 |
| 摘要生成 | 话题背景 | GPT-4/通义 | 中 |
| 向量嵌入 | 语义检索 | text-embedding | 低 |

**成本控制**：
- 每日新增话题约500个
- 每个话题处理约500 tokens
- 每日成本约$0.25
- 每月成本约$7.5

**优化策略**：
- 批量处理：减少API调用次数
- 缓存结果：相同话题不重复处理
- 阈值过滤：只处理热度超过阈值的话题
- 模型选择：简单任务用小模型，复杂任务用大模型

| 营销平台下沉市场 | 竞争加剧 | 保持价格优势、聚焦热搜场景 |
| 用户付费意愿低 | 收入不及预期 | 免费版引流、增值服务、B端优先 |

### 11.3 产品风险

| 风险 | 影响 | 应对措施 |
